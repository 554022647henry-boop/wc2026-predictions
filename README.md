## 一、为什么要做这个

2026年FIFA世界杯是史上规模最大的一届——48支球队、104场比赛。每天都有比赛开踢，每场都有值得讨论的话题。

市面上的预测方式大概分三类：
- **专家经验**：可解释，但主观，无法量化
- **赔率模型**：高效，但赔率本身就含市场偏见
- **ELO/FIFA积分模型**：客观，但只看历史，忽略当下状态

这三类方法有个共同缺陷：**不够结构化，也不够可解释**。你知道一个预测是什么，但很难拆解它"因为什么"这么预测。

我想做一个不一样的系统：**把每场比赛的决定因素拆开来打分，各自独立评估，最后用数学公式聚合**。就像一个专家小组开会，每个专家只说自己熟悉的方向，最后综合投票。

## 二、系统全景

整个系统的预测流水线是这样的：

```
赛程信息
    ↓
[Agent 1：信息搜集]  ← ESPN API + 官方知识库 + 网络资讯
    ↓
[Agent 2：信息审阅]  ← 给每类信息打质量分，不足触发补充搜集（最多2轮）
    ↓
[7个维度 Agent]      ← 各自独立对比赛进行评分
    ↓
[数学聚合模型]       ← 加权分差 → 概率分布
    ↓
[裁判 Agent]         ← 概率 → 预测方向 + 置信度 + 理由
    ↓
[HTML生成]           ← 更新公开网站
    ↓
GitHub Pages         ← 发布，Git时间戳证明赛前预测
```

共涉及 **10个 Agent 角色**，数据、推理、数学、展示四层分离。

## 三、数据基础：真实数据 vs AI 幻觉

这是整个系统最重要的设计决策。

早期版本里，我用 AI 模型直接生成"球队大名单"和"近期战绩"。结果呢——AI 以非常确定的语气告诉我，"法国队的关键球员是格列兹曼"。格列兹曼已经国家队退役了。AI 还给我"编造"了几场根本没打过的比赛结果。

这不是 AI 在说谎，这是语言模型的天然局限：**它的训练数据有截止日期，而球员名单和近期战绩是实时变化的**。

解决方案是：**从 ESPN 官方 API 获取真实数据，绝不用 AI 生成这两类信息**。

系统现在的数据来源分四个等级：

| 数据来源 | 内容 | 可信度 |
|---------|------|--------|
| ESPN API | 26人大名单、近期战绩、赔率、首发、伤停 | ✅ 官方真实数据 |
| WhoScored / CBS Sports | 赛事报道、赛程确认 | ✅ 可信 |
| DeepSeek（AI） | 历史大赛表现、教练风格、战术倾向 | ⚠️ 有幻觉风险，标注不可信 |

这三类数据在系统里有明确的标注（`_source` + `_verified` 字段），不同来源的信息在进入 Agent 推理前会被明确区分。

---

## 四、信息质量控制：Agent 2 的作用

光有数据还不够。在进入预测之前，有一个专门的**审阅 Agent（Agent 2）**负责检查信息完整性。

它的工作逻辑是：

1. 对每类必要信息打 0-10 分的质量分
2. 如果某类信息得分低于 3 分（信息不足或不可信），触发**补充搜集**
3. 最多补充 2 轮，超过则强制结束（避免无限循环）

必须检查的信息类别包括：
- 两队伤停/停赛情况
- 两队近期 3+ 场比赛战绩
- 当前赔率
- 如果距开球不足 1 小时：需要确认首发阵容

只有经过审阅的结构化数据，才会传给后续的 7 个维度 Agent。这一层设计防止了"垃圾进垃圾出"。

---

## 五、7个维度 Agent：核心创新

这是整个系统最重要的部分。

传统的 AI 预测通常让一个大模型综合考量所有因素，给出预测结论。问题在于：这种方式容易产生**"因素叠加"偏差**——如果 AI 先认定了一支队伍更强，后续的所有分析都会被这个先入印象影响，形成自我强化的循环。

解决方案是**分离关注点**：把预测依据拆成 7 个独立维度，每个维度由一个独立 Agent 负责，只评估自己的角度，完全不看其他维度的结论。

7个维度及其权重设计如下：

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| A 绝对实力 | 7% | FIFA排名、阵容身价、是否有超级球星 |
| B 球队化学 | 8% | 核心球员稳定性、新老球员经验分布 |
| C 近期状态 | 7% | 最近5场表现、进球/失球趋势 |
| D 关键球员 | 15% | 核心球员是否缺阵、状态是否在线 |
| E 情境压力 | 20% | 主客场因素、晋级形势、心理压力 |
| **F 战术对位** | **25%** | 阵型克制关系、边路攻防、中场人数对比 |
| G 大赛DNA | 18% | 历史世界杯表现、首战规律、逆境能力 |

权重的设计依据是：在世界杯小组赛阶段，**战术匹配度和情境压力的影响远超纯粹的实力差距**。两支强队之间，决定比赛走向的往往是谁的阵型更能克制对方，以及谁更需要这场赢球。

每个 Agent 使用统一的评分量表：

| 分值 | 含义 |
|------|------|
| 5.0 | 双方完全均等，无明显优劣 |
| 6.0 | 轻微优势（有 1 个有利因素） |
| 6.5 | 明显优势（有 2 个有利因素） |
| 7.0 | 明显占优（3 个以上有利因素） |
| 7.5 | 强势占优（对手无相应反制手段） |
| 8.0 | 压倒性优势（纸面级别的碾压） |

**硬性规则**：双方得分之和必须在 8～13 之间。如果 AI 给出了极端评分（比如两队都打 7 分），系统会自动等比例缩放回合理区间。这一规则防止了 AI "两边都喜欢"的模糊评分。

## 六、数学聚合模型

7个 Agent 各自给出评分后，用数学公式聚合成最终概率。

**关键指标：加权分差 δ**

每个维度计算"A队优势"：用该维度的固定权重，乘以 Agent 的信息置信度，再乘以 A 队与 B 队的得分差。把 7 个维度的贡献加权平均，得到一个浮点数 δ。

- δ > 0 意味着 A 队综合占优
- δ < 0 意味着 B 队综合占优
- δ ≈ 0 意味着双方非常接近，平局可能性高

**概率映射**

δ 通过一张预先校准的查找表，映射为三方概率（A胜 / 平局 / B胜）。这张表参考了历史世界杯的平局率（约 24-27%）进行校准，使系统不会在平局方向上系统性低估。

**赔率校验**

最后还有一个校验步骤：把 ESPN 赔率隐含的市场概率与模型概率对比。如果差距超过 20%，系统会触发 `calibration_warning`，并降低最终置信度。这不会改变预测方向，只是如实标注"模型与市场判断有分歧"。

**置信度判断**

根据 δ 的绝对值和平均信息充分度，最终输出 高/中/低 三档置信度。高置信度意味着多个维度一致指向同一方向且信息充分，低置信度意味着信息不足或维度意见分散。

## 七、裁判 Agent：从数字到表达

数学模型的输出是一组概率和数字，还需要一个裁判 Agent 把它转化为人可读的预测。

裁判 Agent 的任务是：
1. 确认或微调预测方向（极少数情况下会否决数学结果）
2. 用三条具体理由解释为什么这么预测（要求提到具体战术、球员状况，不能是空话）
3. 指出最主要的**爆冷风险**——每场比赛都有可能让预测落空的关键不确定因素

裁判 Agent 同样受到严格的防幻觉约束：**只能提及 ESPN 官方知识库中出现的球员名字**。如果不确定某球员是否在本场名单，用"队长"、"前锋核心"等描述代替，而不是猜一个可能错的名字。

## 八、可解释性与可溯源性

每次预测都会保存完整的内部数据，包括：
- 7 个维度的详细评分和置信度
- 每个维度的关键证据（3 条具体的文字说明）
- 数学聚合的中间过程（δ 值、各维度贡献度）
- 是否触发了赔率校验预警
- 信息审阅的质量评分和补充记录

这些内部数据不面向最终用户，但完整地记录了每次预测的推理链路，方便事后复盘哪个维度判断有误。

---

## 九、Git 时间戳：赛前预测的不可抵赖性

这是一个容易被忽视但很重要的设计细节。

每次预测完成后，系统会以 Git commit 的形式提交到仓库，commit message 包含比赛信息和预测结果。GitHub 的 commit 时间戳是公开可验证的。

这意味着：只要在比赛开始前提交了 commit，任何人都可以查证"这个预测确实是赛前做出的"，而不是事后补写的。

## 十、版本演进：8次关键迭代

从 v0.1 到 v1.0，系统经历了 8 次较大的修复，每次都是踩了真实的坑之后。

### v0.2：从 3 个 Agent 到 7 个 Agent

最初小组赛只用 3 个 Agent。很快发现其中的"大赛DNA"维度在小组赛永远给出中性的 5/5 分，因为小组赛数据太少，Agent 无法判断，就保守地不打分——等于这个维度完全无效。实际上只有战术和情境两个维度在工作，相当于两人投票。

修复：小组赛也启用全部 7 个 Agent，让 DNA 维度在信息充分时正常发挥作用，信息不足时也有 confidence 值体现不确定性。

### v0.4：时间计算 Bug 导致"未来预测"污染现在

系统设计了多轮预测机制（初始预测、赛前24小时、赛前2小时……），用时间差判断触发哪一轮。有一场比赛的触发时间算错了整整 36 小时，导致"赛前2小时预测"在比赛前 38 小时就跑了——而且用的是那个时间点的不完整信息。更糟的是，网站显示逻辑优先展示"更新的预测"，结果 36 小时后的正确预测反而被那个早期错误预测覆盖了。

修复：只保留初始预测用于展示，删除了 15 个被污染的预测文件，重新跑了受影响的比赛。

### v0.5：信息与维度严重错位

战术维度 Agent（F_tactical）收到的信息里写着"法国队采用 4-3-3"——这是阵型名称，不是战术分析。Agent 根本无法用这条信息判断"4-3-3 克不克制对方的阵型"。

问题在于：搜集信息的 Agent 不知道预测 Agent 需要什么样的信息，两者语言不对齐。

修复：重新设计了知识库格式，明确要求包含"进攻模式"、"防守薄弱点"、"克制关系"等可用于对比的结构化字段。同时建立了统一评分量表，防止 AI 自由发挥评分尺度。

### v0.6：AI 生成知识库替换为 ESPN 真实数据

这是最根本的一次改造，起因是预测理由里出现了格列兹曼的名字。他早就不在法国国家队了。

根源：知识库 v1 里的球员名单全部由 AI 生成，AI 的训练数据截止日期不知道，它会给出它"认为"应该在名单里的球员。

修复：用脚本从 ESPN API 拉取 48 支球队的官方 26 人大名单，每条记录标注 `ESPN_ROSTER ✅`，覆盖所有 AI 生成的大名单。不确定的字段填 null，不填假数据。

### v0.8：评分方向倒置

有一场 Jordan vs Argentina 的预测，维度 Agent 给出了 Jordan 更强的评分。Jordan 是本届世界杯最弱的队伍之一，Argentina 是卫冕冠军。

排查后发现：Agent 的 prompt 里写的是"请评估两队的相对优势"，但没有明确"team_a 对应哪支队"。AI 理解混淆了左右，把 Argentina 的优势评给了 Jordan。

修复：在所有 Agent 的输入里明确标注"xxx = team_a（左边队），yyy = team_b（右边队），如果 yyy 更强则 team_b_score 更高"，消除歧义。

## 十三、已知局限与下一步计划

**当前系统的主要局限**：

1. **平局识别偏弱**：系统倾向于预测有明显优势的一方获胜，在双方实力接近时识别平局的能力还有提升空间
2. **球员状态数据粒度不足**：ESPN API 提供名单，但不提供球员在俱乐部本赛季的详细数据（进球数、场均评分、伤病频率等）
3. **淘汰赛逻辑尚未充分测试**：小组赛阶段的逻辑已经跑通，16强之后的淘汰赛需要额外的加时赛/点球大战处理

**下一步计划**：
- 接入 Sofascore 或 FBref 的球员赛季数据，丰富 D_keyplayer 维度的输入质量
- 优化数学模型中的平局区间判断，提升平局识别率
- 在小组赛结束后，进行系统性的复盘，重新标定各维度权重
